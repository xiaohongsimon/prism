## 1. 一句话

Prism 是作者本人的跨渠道 / 跨语种 / 跨模态信息工作台。服务作者本人是唯一目标。

## 2. 主用户

**只有一个：作者本人。** 任何设计冲突下，作者的使用体验优先。不存在次级用户、目标受众、"顺带受益者"在 mission 层的地位。

## 3. 识别的四大痛点

1. **跨渠道**：X / YouTube / HN / arXiv / GitHub / 小宇宙 / Reddit / PH / ... 聚到一起
2. **跨语种**：本地 AI 翻译外文内容，快速筛选
3. **跨模态**：视频 / 播客 / 长文 → **两条分叉的消费路径，不共享加工管线**：
   - **人类消费路径**：结构化 tl;dr + 观点 + 章节高亮；压缩 + 过滤，为的是在时间稀缺下快速筛选
   - **作者 agent 消费路径**：原文 + 轻量元数据（source / author / published_at / url / 可选章节 offset），**不压缩、不过滤、不二次加工**。Prism 对 agent 的价值是**聚合 + 检索**而非转换；压缩过的内容对 agent 是损失而非节省，agent 上下文大、下游推理吃的是全量。这条路径在人类加工管线**之前**分叉，直接暴露 `raw_items.body` 级别的原文
4. **信息过载 × 单渠道推荐的天花板**：X 的推荐只看到我在 X 上的行为，YouTube 只看 YouTube；任何单渠道平台都只拥有我偏好的碎片。**Prism 因覆盖全渠道（痛点 1），天然拥有任一单平台拿不到的完整偏好视图**——长期目标是成为比任何单渠道平台更懂我的信息推荐官

**当前聚焦**：痛点 1-3（跨渠道抓取 + 翻译 + 结构化是痛点 4 的前置数据基础）。痛点 4 的解决方案见 §8"延后但未放弃"

## 4. Non-negotiables（硬约束，改动前必须回到本节辩论）

| # | 约束 | 理由 |
|---|---|---|
| **NN1** | 个人工作台定位：冲突下偏作者 | 初心；非多用户 SaaS 的结构性决定 |
| **NN2** | 本地算力是一等硬上限（≤ Mac Studio 容量） | 决定 active 源数量天花板，拒绝无节制扩张 |
| **NN3** | **偏好数据是隐私**：画像 / 排序结果 / 学到的权重 / 可反推个人偏好的模型产物（embedding、特征字典、微调权重）**不入 git、不对外呈现**；实现代码可公开。架构上隔离到 `prism/personalize/`，数据只存本地 DB | 偏好即画像即身份暴露面。代码开源可以，画像不行 |
| **NN4** | 源和历史内容**降频不删除** | Prism 是档案库不只是 feed；出局 = 降频，不是清理 |
| **NN5** | 作者 agent 访问本地内容是一等场景 | 作者的 agent（Claude Code / 本地脚本 / 其他本机 AI 工具）读 Prism 的 article / feed / search 属于"作者本人使用"，不是次级用户。接入形式可以是 CLI、MCP server、本机 HTTP 只读端点；形式选择是实现细节，"作为一等场景存在"是 mission 级承诺 |
| **NN6** | **敬畏上线**：所有自动化写入用户可见路径的行为（LLM 产物、source 自动上下线 / 优先级变更、pipeline 自动决策影响 UI 呈现）必经质量门禁，不过关不上线。任何新 pipeline 上线前必须定义：**(a) 作者一键标记失败的反馈入口；(b) 连续失败即降级为 staging 的阈值**。具体门禁机制与 shadow/draft 架构见 `tech-stack.md §6`，不接受静默降级 | 认知污染比代码 bug 更可怕——错误翻译、幻觉摘要、被悄悄拉进 feed 的低质源，作者吃下去还不自知。"跑什么就显示什么"不符合 Prism 的基本态度 |
| **NN7** | **拒绝多用户 / 多租户**：代码开源但**架构、issue 响应、文档不以多用户为前提**；任何引入权限隔离 / 租户模型 / 共享部署假设的 PR 默认拒绝。开源是副作用，不是承诺 | 单人项目最大的腐蚀不是技术债而是社交压力——朋友求账号、issue 求多租户支持、PR 加权限系统。没有硬红线，NN1（个人工作台）会被逐步稀释 |

**NN 冲突优先级**：当 NN3（隐私）与 NN6（质量）冲突时，**NN3 > NN6**。当 NN3 / NN6 与任何产品收益（包括但不限于推荐效果、吞吐、响应速度）冲突时，NN 优先，不得以产品收益为由稀释（典型诱惑：匿名化偏好上报、关闭门禁以提高吞吐等）。

### NN3 在 agent 场景的分级（2026-04-24 明文化）

适用范围：Prism 通过 CLI / MCP / 本机 HTTP 被本机 agent 读取的场景。

- **算内**（不触 TW3）：作者主动触发的单次会话；payload 只含 article 内容本身（`raw_items.body` 原文 / `articles.structured_body` 人类路径产物）。典型：你在 Claude Code 里问"最近看了啥"，agent 调 `prism feed --format ai` 或 MCP 取若干条 article 给你分析。**注**：agent 路径默认只吃 `raw_items.body` + 元数据（§3.3 分叉原则），结构化产物仅在 agent 明确请求时附带
- **算外**（触 TW3）：payload 里带 `preference_weights` / `personalize/` 产物 / 排序结果 / 任何可反推画像的派生物；**或**自动化定时任务把全量 article 批送 cloud LLM 做深加工
- **灰区需登记**：任何 batch（单次调用涉及 > 20 篇 article）且走 cloud LLM 的消费必须在 `decision_log` 留记录（调用方 / 时间 / 范围 / 目的）

**判定口径**：NN3 保护的是**偏好**，不是**"作者看过的内容"**。Article 内容本身（公开信号源 + 本地加工产物）不是偏好；偏好是"作者选择看哪些、给哪些打了正反馈、在 `/feed` 里怎么排"。TW3 触发条件以此为准。

### NN7 在 agent 场景的分级（2026-04-24 明文化）

NN5 承认"作者 agent 访问本地内容是一等场景"，但这条**不等于**"任何被授权的 agent 都是一等场景"。两条 NN 在 agent 场景的边界如下。

**三问判定**（对任一 agent 接入，三个都为"作者本人"进 NN5；任一为他人进 NN7）：
1. 机器是谁的？（Prism 跑在谁的物理硬件上）
2. 触发者是谁？（这次调用是谁敲的 prompt / 跑的脚本）
3. OS 账号是谁？（连接所用的用户身份）

**判定结果**：

- **NN5 算内**（允许，是一等场景）：作者本人触发、跑在作者物理控制的机器上、以作者 OS 账号身份读写的 agent。例：你自己的 Claude Code 跑在 Mac Studio 上读本机 :8080 或本机 MCP；你本地脚本 cron 跑 `prism feed --format ai`
- **NN7 禁止**：作者通过 token / SSH / 外网 MCP / 授权凭证把 Prism 暴露给**他人**运行的 agent，不论对端是朋友的 Claude Code、朋友的 Cursor、还是朋友的脚本。理由：一旦合法化，Prism 必须开始维护 token 生命周期 / scope / 撤销 / per-token 访问审计——这正是 NN7 守的"多用户工程"边界，与 NN1（个人工作台）结构性冲突
- **NN3 额外触发**：即便技术上允许对端 agent 连接，只要 payload 含偏好产物（`preference_weights` / `personalize/` / 排序结果），在走出作者物理机器的那一刻起 TW3 触发

**若将来真需要跨主体 agent 访问**：不走 NN7 例外通道。走 ecosystem charter 辩论（`docs/reviews/synthesis/2026-04-24-mission-v8-ecosystem-agenda.md`），与 Prism → wechat-insight 消费议题合并评估。Prism 本体不为此提供一等接入。

## 5. 源的选入与活跃度

- **选入**：当前主要靠作者人肉添加。X 推荐流 / follow-builders 等作为辅助参考信号，但不是主流；系统化的"自动选入机制"暂缺，承认这是未解问题
- **暂缺阻塞锚点**：在 §8 个性化推荐引擎开始消费 `/feed` 偏好反馈作为训练数据之前，系统化选入机制必须**就位**；否则推荐引擎学到的就是人肉选源偏见。暂缺状态若阻塞了 §8 启动，强制触发 §9 mission 重评
- **"就位"最低验收口径**（防止无限延期 / 自我豁免）：以下五要素闭环即视为就位，不要求"全自动选入"：
  1. **候选源生成**：存在非人肉驱动的候选源产出通路（LLM 基于偏好画像推断 / 从关注图谱延展 / 从外部引用反查任一种）
  2. **质量理由**：每个候选源带机器可读的推荐理由（源类型 / 内容采样 / 与现有源的偏好相似度等）
  3. **作者 approve/reject**：作者对候选源的决定通过 UI / CLI 落 DB，不走口头或 git commit
  4. **`decision_log` 记录**：approve/reject 都写入 `decision_log`，可追溯
  5. **降频不删除**：reject 的源进入低优先级长窗口，不从 `sources.yaml` 物理删除（与 NN4 一致）
- **活跃度分层**：算力有限约束下，不同源有不同优先级 + 抓取时间窗口（高优先级 → 高频抓；低优先级 → 低频 / 长窗口）。不存在显式的"出局 / 淘汰"动作
- **降频 ≠ 删除**（NN4）：优先级下降 = 时间窗口拉长，数据和源本身永久保留

## 6. Usage model

| 层级 | 路径 | 作用 |
|---|---|---|
| **主路径**（日用） | `/feed/following` | 订阅更新流 |
| **主路径**（日用） | `/creator/{key}` | 单博主全量历史 |
| **主路径**（日用） | `/article/{id}` | 视频/长文结构化阅读（人类消费形态） |
| **主路径**（agent） | `prism ... --format ai` / 本机 MCP / 本机 HTTP 只读 | 作者 agent 消费原文 + 轻量元数据；**与人类路径分叉**，不吃结构化产物（见 §3.3）。形态细节在 `tech-stack.md` |
| **战略路径**（长期关键） | `/feed` | 扩大知识源 + 积累偏好反馈——未来个性化推荐引擎的训练数据入口。此处积累的显式反馈受 NN3 约束，仅用于本地模型训练，不对外呈现 |
| 次路径 | `/briefing` | 周期性回顾 |

Source 维护、运维仪表等作者专用路径属于实现细节，由 `tech-stack.md` / `roadmap.md` 定义。

## 7. 当前优先投入

1. **技术债务清理优先于新功能**——历史定位残留不得阻塞当前主路径体验（具体清单见 SPEC §6 §11 和 roadmap Wave 1）
2. 翻译质量 + 时效
3. 结构化质量（视频转文字 / tl;dr / 观点抽取，人类路径）+ 作者 agent 的原文通路（CLI / MCP，见 §3.3 分叉原则）
4. "新更新"状态（`/feed/following` 的核心属性，非独立功能）
5. 订阅管理 UX + 低价值订阅建议
6. 原文链路可达性

## 8. 延后但未放弃（not-now，非 non-goal）

- **数据源的自动探索 / 发现 / 引用机制**（§3 痛点 1 的长期命题）——跨渠道覆盖当前靠人肉拓源；系统化的"自动探索（从关注的人关注谁、从热门话题反查原始作者、从被引用的外部链接回溯源）+ 质量判断 + 自动选入"机制仍**无解**。此命题比推荐引擎更底层：推荐引擎至少知道目标函数（偏好匹配），源发现连"什么是好源"都未定义清楚。承认这是硬骨头，不假装有方案；与 §5 选入机制暂缺同源
- **偏好推荐引擎（§3 痛点 4 的解决方案）**——镜子而非投其所好；比任一单渠道平台更懂作者，靠的是跨渠道偏好视图（痛点 1 的副产品）而非更强的排序模型。在 `prism/personalize/` seam 里重建，取代已清理的 BT/CTR。启动前置条件：满足 §5 五要素闭环；排序输出受 NN6 约束；训练数据受 NN3 约束；冲突时遵守 §4 优先级声明。**未满足 §5 就位前**：只允许离线实验和 shadow 排序（双轨运行、对比评估），**不得影响 `/feed` 可见排序**

## 9. Trip-wires（触发 mission 重评或自动熔断的事件，月度自动统计）

| # | 触发条件 | 行动 |
|---|---|---|
| **TW1** | 作者连续 **14 天**未访问 `/feed/following` | **熔断**：暂停 pipeline 调度，防止算力空转和数据腐烂；重新使用时手动恢复 |
| **TW2** | 本地算力天花板被突破（推理队列积压 > 72 h） | 必须缩源、调低源优先级、或涨机器，不得静默降质 |
| **TW3** | **NN3 配套**：偏好数据目录 `prism/personalize/data/` 或任何可反推画像的产物出现在 git diff / 外部错误上报（Sentry 等）/ 公开 dashboard / 对外日志 | **立即熔断**该外流渠道，审计泄漏路径并回填到 tech-stack 保密清单 |
| **TW4** | **NN6 配套**：2 周滚动窗口内某条 pipeline 累计被作者事后标记为"污染" ≥3 次，或同类错误（同一种翻译幻觉模式 / 同一类摘要失真）≥2 次 | 暂停该 pipeline 的自动写入，改 staging + 人工 approve 模式，直到门禁机制修复并观察 2 周。单次污染必须立即回撤该产物并落 `quality_anomalies`。污染标记入口见 `tech-stack.md §6.3`；**统计阈值以本条为准**，tech-stack 可按风险等级声明更严的局部阈值（如 High 风险管线），但不得保留全局 ≥1 触发 |
