# Prism — Codex Review

> Reviewer / challenger 视角的设计审查意见

## 1. Review Summary

当前 Prism 仍处于 design-first 阶段，仓库内尚无实现代码。就设计稿本身而言，方向是清晰的：通过减少源数量、增加单源深度、引入聚类和日级批量分析，来替代 `shu` 的逐条摘要模式。

但从 reviewer 视角看，当前设计里最需要优先收敛的，不是技术选型，而是几个会直接影响成败的关键假设：是否真的能获得“时间差”、聚类质量如何验证、分析结果如何支持重算与审计、以及配置/运行态之间的冲突如何收口。

## 2. Findings

### [P0] 目标与手段之间存在张力：当前 v1 方案未必能兑现“更早看到信号”

设计稿把“比公众更早看到信号”作为核心问题之一，但 v1 的主要源仍然是精选 X 头部账号、arXiv 和 GitHub Trending。这些源本身依然高度公开，且大多已经处于信息扩散链的中后段。相比 `shu`，Prism 更可能提升的是“信噪比”和“解释深度”，而不是显著提升“时间优势”。如果不先把 success criteria 从“更早”改成“更准、更可行动”，后续很容易出现系统按设计实现了，但目标仍被判定为未达成的情况。

### [P0] `merged_context` 的截断规则依赖后置字段，当前设计存在自洽性问题

设计稿写到，cluster 内上下文超长时“优先保留 `signal_strength` 高的 item”。但 `signal_strength` 是第二阶段 LLM 分析后才会产出的字段，不属于预处理阶段可用信息。也就是说，当前规则在执行顺序上无法成立。这里需要明确替代排序依据，例如 source priority、freshness、引用次数、repo 增速、或增量阶段的粗打分结果。

### [P1] X thread 深采集是系统价值点，也是最脆弱的外部依赖

Prism 与 `shu` 拉开差距的重要手段之一是 thread 展开和引用链抓取，但方案依赖 syndication + playwright 的组合。这个链路天然容易受到页面结构变更、限流、登录态和反爬策略影响。一旦 thread 展开不稳定，系统会迅速退化成“精选账号首条推文摘要器”，那就很难支撑“深挖掘”的定位。设计里虽然有 `thread_partial=true` 的 fallback，但还缺少对 thread 完整率、降级比例和源级质量告警的要求。

### [P1] 聚类方案能快速起步，但误合并/漏合并风险较高，且缺少评估闭环

v1 聚类算法以 URL、repo 名、标题相似度和实体共现为主，工程上是合理的第一步。但这些规则对 AI 领域高频话题并不稳健：同一技术概念可能被不同表述拆散，不同事件也可能因为共享实体被错误并簇。当前设计没有定义任何人工抽样评估、precision/recall 代理指标，或“错误聚类如何纠正”的机制。没有这层闭环，后续 LLM 批量分析质量会高度依赖前置聚类偶然性。

### [P1] `signals` 存储设计不足以支撑覆盖式重算、版本比较和审计

设计明确区分 hourly incremental 与 daily full analysis，并说明 daily 会覆盖当天 incremental 结果。但表结构描述里只列出了 `signals` 的业务字段，没有看到 `analysis_type`、模型版本、prompt 版本、生成时间、是否当前生效等关键元数据。少了这些字段，后续会很难回答“今天这条判断是哪个模型在哪次任务里生成的”“为什么日级结果和小时级结果不一致”“能否重放昨天的分析”。

### [P1] 三套源管理入口并存，但冲突仲裁规则还不够明确

当前设计同时支持 YAML 声明式配置、SQLite 运行时状态、CLI/API 动态 CRUD。这个方向很灵活，但也引入了 source identity 和 ownership 问题。例如：同一个 source 是用 `(type, handle)` 还是稳定 UUID 识别；CLI 动态新增的 source 是否应回写 YAML；YAML 删除导致 DB disabled 后，CLI 再启用是否允许；API 更新和下次启动 reconcile 冲突时谁胜出。设计里给了高层原则，但没有写出可执行规则，后续实现时很容易出现状态漂移。

### [P2] 自动禁用策略偏激进，可能把“临时外部波动”误判成“长期坏源”

“连续 3 次失败自动禁用该源”在实现上很简单，但不同源的失败语义并不一致。对于 X 和 GitHub 这类强依赖外部页面与限流状态的 adapter，短时失败并不一定意味着源失效。若没有区分失败类型、恢复窗口和人工复位机制，这条策略可能把本来高价值但偶发不稳定的源过早下线，反而损害整体质量。

## 3. Open Questions

1. Prism 在 v1 的首要价值，到底是“更早发现”，还是“更高信噪比 + 更强关联 + 更可行动”？
2. 聚类质量准备怎么验收？是否接受先做 1 周人工标注抽样，再决定是否需要 embedding 或 reranker？
3. daily analysis 覆盖 incremental 时，历史结果是物理覆盖、逻辑失效，还是版本并存？
4. source 的唯一标识和最终真实来源是什么？YAML、DB、CLI/API 三者谁拥有最终裁决权？
5. X thread 展开如果长期不稳定，Prism 的最小可用产品是否仍成立？如果成立，差异化还剩什么？

## 4. Suggested Next Decisions

1. 先重写 success criteria，把“时间差”降为待验证假设，把“质量提升”和“可行动性提升”设为首要验收目标。
2. 先补全 data model，至少把 source identity、analysis provenance、signal versioning、job run metadata 定义清楚。
3. 先定义聚类评估办法，再决定规则法是否足够进入 v1。
4. 先把 X adapter 的退化路径、观测指标和告警阈值写出来，避免实现后才发现主价值点不稳定。
5. 在真正开始写代码前，先补一页“state machine / ownership”文档，专门说明 YAML、DB、CLI/API 的关系。

## 5. Reviewer Stance

后续如果 cc 开始实现，我会优先从以下角度做 review：

- 是否把“高层设想”落实成了可验证的数据模型和任务边界
- 是否引入了隐藏状态或不可重放逻辑
- 是否对外部依赖失败、部分成功、重复运行和回填场景做了足够设计
- 是否存在先做复杂分析、后补观测与评估的顺序问题

当前结论很简单：Prism 的方向值得做，但在开始大规模实现前，需要先把“目标定义、聚类评估、分析版本化、配置 ownership”四个基础问题钉牢。
